# 三轮审计总结

审计日期：2026-03-19

## 审计范围

本次审计覆盖：

- `01-analysis-report.md`
- `02-comparison-table.md`
- `03-initial-implementation-plan.md`

## 第 1 轮：事实与来源审计

### 发现的问题

- 报告里有一处内部矛盾：文字写成了“拆成两个产品”，但实际分析已经按三层域来展开。
- 几个 exporter 类项目的能力描述偏乐观：
  - `node_exporter` 容易让人误解成直接提供现成的流量速度和磁盘速度，但很多场景下这其实是基于 counter 派生出来的。
  - `ntopng` 需要补充 edition 差异说明，避免把高阶能力默认成社区版现成能力。
- 原始文档对一个关键安全点强调不够：把 exporter 端口直接暴露公网，不应作为默认接入方式。

### 已完成修复

- 把结论统一成 `主机指标`、`网络质量`、`流量分析` 三层。
- 给 `node_exporter` 补了派生速率说明。
- 给 `ntopng` 补了 edition 差异提醒。
- 在分析报告里补了私网、隧道、鉴权网关的接入基线。

## 第 2 轮：对比逻辑审计

### 发现的问题

- 原始表格把 `历史/告警` 合在一列，掩盖了真实差异。
- 原始表格把 `网络波动` 和 `主动探测` 混在一起，导致被动流量工具和主动探测工具看起来过于接近。
- 一些打分过宽：
  - `Glances` 的每核心能力被写弱了。
  - `node_exporter` 的流量和磁盘速率能力需要从“现成功能”降到“可派生能力”。
  - `Cockpit` 的告警能力需要更保守。
  - `Nagstamon` 在客户端侧不应被默认视为带历史能力。

### 已完成修复

- 把 `历史` 和 `告警` 拆成独立列。
- 把 `主动探测` 和 `流量统计` 拆开。
- 重调了多处项目打分，让结果更保守也更接近实际实现成本。
- 增加审计备注，明确某些列代表“可派生能力”，不是“开箱即用的现成页面”。

## 第 3 轮：架构与安全审计

### 发现的问题

- 实现方案里对 exporter 暴露方式的安全约束不够明确。
- 方案没有明确指出：桌面端发起的探测，只能代表 operator 路径，不代表服务器侧路径。
- 方案对 counter reset、staleness、资源身份漂移这些数据正确性问题写得不够细。
- 方案对并发抓取、批量写入、历史清理这些运行期保护写得不够细。

### 已完成修复

- 新增 `安全接入基线`。
- 新增 `Probe 视角模型`。
- 新增 `指标语义守卫`，覆盖 counter/gauge/reset/staleness。
- 新增 `运行期保护`，覆盖并发上限、批量写入、保留清理。
- 收紧 Phase 1，明确首套部署必须基于私网或隧道，而不是直接抓公网 exporter。

## 当前仍保留的风险

- 文档仍然是 Linux-first 的 MVP 假设。
- v1 的流量分析仍是轻量版，不替代完整网络可观测性产品。
- 如果未来要支持多桌面共享历史或多用户协作，还需要单独做一轮架构评审。

## 审计结论

- 状态：`已增强`
- 结果：当前这组调研文档比原始版本更准确、更保守，也更适合作为后续实现决策依据。
